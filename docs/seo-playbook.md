# SEO-регламент Гермеса — как создавать и оформлять страницы

Обязателен к прочтению **перед созданием или правкой любой посадочной/статьи**. Цель: расти в поиске без каннибализации, без дублей, без гонок в git. Источник данных для контента — только `data/*.json` (это OSINT-оценка, не официальные данные; дисклеймер обязателен).

Канон: одна страница = один поисковый интент. Правило приоритета — [[docs/npz-posting-style-v2]] для постов, этот файл для веб-страниц.

---

## 0. Железное правило: сначала реестр, потом страница

Перед созданием страницы **открой `data/seo-topics.jsonl`** и проверь пересечение ключей.

- Если ≥40% целевых ключей новой темы уже покрыты существующей страницей → **НЕ создавай новую**. Дополни существующую или выбери более узкий интент.
- Если создаёшь — **сразу добавь строку в `data/seo-topics.jsonl`** (одна JSON-строка на страницу, см. схему внизу) в том же коммите.
- Не бывает двух страниц под «нет бензина», двух под «дефицит», двух под «топливный кризис». Это разные формулировки ОДНОГО интента — им нужна одна каноническая страница, остальные ссылаются на неё.

Каждая тема попадает в один из ПЯТИ типов, они НЕ конкурируют между собой. **`type` в реестре напрямую задаёт группу в меню «Аналитика»** (см. §7):
1. **Регион** (`region`) — `/crimea`, `/<region-slug>` (интент «топливо/бензин в <регионе>») → группа **Регионы**.
2. **Объект** (`object`) — `/npz/<slug>` (биография НПЗ) → в меню НЕ показывается, доступ только через `/refineries`.
3. **Объяснялка** (`explainer`) — `/deficit`, `/talony` (интент «почему/что значит») → группа **Объяснялки**.
4. **Прогноз** (`forecast`) — `/crisis` (интент «когда/сценарии/что дальше») → группа **Прогноз**. НЕ путать: `/deficit` = причины, `/crisis` = прогноз.
5. **Справочник** (`reference`) — `/refineries`, `/attacks` (интент «список/хроника всего») → группа **Справочники**.

## 1. Анатомия страницы (обязательный минимум)

```
<title>            — 50–60 знаков, главный ключ В НАЧАЛЕ. Пример: «Дефицит бензина в Крыму 2026: АЗС, цены, талоны»
<meta description> — 150–160 знаков, главный ключ + цифра/факт + призыв. Не дублирует title дословно.
<meta keywords>    — 5–8 ключей, все реально используются в тексте.
<link canonical>   — self-canonical на чистый URL (без .html).
<h1>               — ровно один, содержит главный ключ дословно.
Лид                — 2–3 предложения, главный ключ в первом, ответ на интент сразу (не «вода»).
H2-секции          — 3–6 штук, каждая закрывает под-запрос; в подзаголовках — вторичные ключи.
Данные             — конкретика из data/*.json: цифры, даты, статусы. Никаких выдуманных чисел.
FAQ                — 6–8 вопросов (объяснялка/регион) или 3–4 (объект). Вопросы = реальные长-tail запросы.
Перелинковка       — ≥3 внутренние ссылки (карта + смежная посадочная + свежая сводка/дата).
Дисклеймер OSINT   — обязателен внизу.
Дата обновления    — «Обновлено <дата> МСК», обновляй при правке данных.
```

**Оформление (тёмная тактическая тема):** переиспользуй классы из `news.css` (`.faq-list/.faq-item`, `.timeline-list/.timeline-item`, `.status-grid/.status-tile`, `.fact-grid/.fact-tile`) и переменные `styles.css` (`--surface2`, `--crit`, `--amber`, `--teal`, `--disp`, `--mono`). Не изобретай инлайн-стили. Эталоны верстки: `crimea.html`, `npz/omskij-npz.html`, `docs/design/news-post-v2.html`.

## 2. Schema.org (JSON-LD, обязательно валидный)

Каждая страница несёт **три** блока, валидируй `json.loads` перед коммитом:
- `Article` (или `NewsArticle` для датированных сводок) — headline, datePublished, dateModified, author, publisher, image, inLanguage:"ru", isAccessibleForFree:true.
- `FAQPage` — все вопросы страницы (Google даёт rich-result, занимает 2–3 позиции на мобиле).
- `BreadcrumbList` — путь (Главная → раздел → страница).

Для страницы объекта (`/npz/*`) дополнительно уместен `Place`/`Organization` с гео.

## 3. URL и файлы

- Регион: `<region-slug>.html` → чистый URL `/<region-slug>` (cleanUrls включён). Slug латиницей: `crimea`, `krasnodar`, `rostov`.
- Объект: `npz/<slug>.html` → `/npz/<slug>`. Slug латиницей: `omskij-npz`, `ryazanskij-npz`.
- Одна тема — один файл. Не плоди `krym.html` рядом с `crimea.html`.
- Добавляй новую страницу в `sitemap.xml` с `lastmod` = дата создания и приоритетом: регион/объяснялка высокочастотная 0.8–0.85, объект 0.7, справочник 0.6.

## 4. Анти-каннибализация — текущие конфликты (ИСПРАВИТЬ)

⚠️ **`/deficit` и `/crisis` каннибализируют друг друга.** Оба целятся в «топливный кризис россия 2026», «когда закончится», «дефицит бензина», «нет бензина». Google будет метаться между ними и просаживать обе.
**Решение:** `/deficit` — канон кластера «почему нет бензина / дефицит» (объясняет причины и масштаб). `/crisis` — либо 301-редирект на `/deficit`, либо перепрофилировать в узкий интент «когда закончится» (прогноз/таймлайн из `data/forecast.json`) без пересечения ключей «дефицит/нет бензина». Не держи обе как есть.

Остальные страницы различимы: `/crimea` (регион), `/attacks` (хроника ударов), `/refineries` (список НПЗ), `/npz/omskij-npz` (объект) — конфликтов нет.

## 5. Git-гигиена (защита от гонок с другими агентами)

На репо параллельно работают несколько исполнителей — коммить дисциплинированно:
- **Перед push всегда** `git pull --rebase --autostash`, затем push. Не форсируй.
- Один логический коммит = одна тема. Не смешивай SEO-страницу с правкой данных сборщиков.
- Файлы `index.html/styles.css/app.js/radar.html/version.json/CHANGELOG.md` защищены pre-commit хуком — трогай только осознанно, с `ALLOW_FRONTEND_RELEASE=1`, и только если правка UI действительно нужна (для новой посадочной они НЕ нужны).
- `sitemap.xml` правят многие — при конфликте мёржи оба набора URL, не затирай чужие.

## 6. Ритм и метрики

- Не больше 1 новой посадочной за прогон — качество важнее количества. Google не любит всплески тонкого контента (YMYL-тематика особенно).
- Раз в неделю — реоптимизация: обнови данные/даты на топовых страницах (`/crimea`, `/deficit`), проверь позиции в Search Console.
- Перед публикацией самопроверка: главный ключ в title+H1+первом абзаце? ≥3 внутренних ссылок? 3 валидных schema-блока? Строка в `seo-topics.jsonl` добавлена? Дисклеймер на месте?

## Схема `data/seo-topics.jsonl` (одна строка = одна страница)

```json
{"url":"/crimea","type":"region","h1":"...","primary_kw":"нет бензина в крыму","keywords":["бензин в крыму","дефицит топлива в крыму","АЗС Крым"],"created":"2026-07-07","status":"live"}
```

Поля: `url` (чистый), `type` (**region|object|explainer|forecast|reference**), `primary_kw` (один главный запрос), `keywords` (список вторичных), `created`, `status` (live|redirect|draft). Перед новой страницей грепай этот файл по своим ключам — если пересечение ≥40%, тема занята.

## 7. Навигация генерится из реестра — руками меню НЕ трогать

Меню «Аналитика» на ВСЕХ статьях/лендингах собирается автоматически из `data/seo-topics.jsonl` скриптом `agents/build-nav.py`. Группа берётся по полю `type` (см. §0). Это значит:

- **Добавил страницу** → добавь строку в реестр (§0) → **запусти `python3 agents/build-nav.py`** → закоммить изменённые HTML. Пункт сам появится в нужной группе на всех страницах. Забыть добавить в меню руками теперь невозможно — не редактируй `<nav class="news-nav">` вручную, перезатрётся.
- Красивую подпись/эмодзи пункта задай в словаре `LABELS` внутри `build-nav.py` (иначе возьмётся `primary_kw`).
- Проверка целостности: `python3 agents/check-ia.py` — падает, если у `status:live` страницы нет файла, записи в sitemap или места в меню. Гоняй перед пушем.
- URL по типам: регион `/<slug>`, объяснялка/прогноз `/<тема>`, завод `/npz/<slug>`, справочник `/refineries`,`/attacks`, сводка `/news/ГГГГ-ММ-ДД`. Дневные сводки НЕ дублировать в корне репо — только в `news/`.
