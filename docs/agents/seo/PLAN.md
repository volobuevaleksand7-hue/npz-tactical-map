# SEO-система «агент агентов» — план и устройство

Дата: 2026-07-09. Владелец: Серёга. Статус: утверждён к запуску.

Цель: сайт растёт → превратить разрозненные SEO-действия (генераторы страниц,
реопты, сводки) в управляемую систему из ролей-агентов с еженедельной петлёй
обратной связи. Каждая роль **переносима**: сегодня её исполняет Claude на Маке,
завтра — Гермес на VPS, Codex или Gemini. Бриф один и тот же.

---

## Принцип переносимости (главное)

- **Роль ≠ платформа.** Каждый агент описан одним markdown-брифом
  `docs/agents/seo/BRIEF_<роль>.md`: роль, входы (файлы), выходы (файлы),
  шаги, проверки, запреты. Исполнитель подставляется в момент запуска.
- **Механика — в скриптах**, не в головах моделей: генераторы, валидаторы,
  реестр уже есть в `agents/`. Агент читает/решает/пишет файлы и зовёт скрипты.
- **Обмен между ролями — только через файлы в репо** (реестр, бэклог, отчёты).
  Никаких «передай в чате»: любой исполнитель видит всё состояние из git.
- 🔴 **Секреты не покидают Claude/Гермеса.** Роли с доступом к аккаунтам
  (GSC, Яндекс.Вебмастер/Метрика под richardsonbihoeki) не делегируются
  DeepSeek/MiMo/Gemini/Codex. Роли по открытым данным — делегируются свободно.

## Файлы системы

| Файл | Что это | Кто пишет |
|---|---|---|
| `data/seo-topics.jsonl` | реестр страниц (анти-каннибализация) — уже есть | писатель |
| `data/seo-backlog.jsonl` | бэклог тем-кандидатов (НОВЫЙ) | разведчик → оркестратор |
| `docs/agents/seo/reports/<YYYY>-W<NN>.md` | недельный отчёт по позициям | аналитик |
| `docs/agents/seo/BRIEF_*.md` | брифы ролей | владелец/оркестратор |
| `docs/seo-playbook.md` | правила создания страниц — уже есть, закон | — |

Схема строки `seo-backlog.jsonl`:
`{"topic": "...", "keys": ["..."], "freq_est": 0, "type": "region|object|explainer|forecast|reference", "status": "idea|approved|in_work|done|rejected", "source": "wordstat|gsc|competitor", "added": "YYYY-MM-DD", "note": ""}`

## Пошаговый план запуска

### Шаг 0 — каркас (сегодня, этот коммит)
Брифы 6 ролей + этот план в `docs/agents/seo/`. Пустой `data/seo-backlog.jsonl`
с первыми темами из уже известного (радар-кластер из `docs/tz-radar-cluster-2026-07.md`,
конфликт `/deficit` vs `/crisis` из playbook §4).

### Шаг 1 — глаза: аналитик позиций (первая неделя)
1. Проверить доступы: GSC по `docs/agents/gsc-setup.md`, Яндекс.Вебмастер
   (мета-тег подтверждён, аккаунт richardsonbihoeki), Метрика 110490245.
2. Прогнать бриф аналитика вручную → первый baseline-отчёт
   `docs/agents/seo/reports/2026-W28.md`: топ-запросы, страницы на позициях
   5–15 (кандидаты на реопт), провалы, динамика кликов.
   Без этого отчёта остальные роли работают вслепую — поэтому он первый.

### Шаг 2 — первый ручной цикл (та же неделя)
Оркестратор (сессия Claude с владельцем) читает отчёт + бэклог и назначает:
1 реопт из списка аналитика + 1 новую страницу из бэклога. Писатель и
реоптимизатор исполняют по брифам. Цель шага — убедиться, что цикл живёт
на файлах, а не на памяти одной сессии.

### Шаг 3 — ритм на автомате (вторая неделя)
- Понедельник: scheduled-task «аналитик» → отчёт; следом сессия оркестратора.
- Вт–пт: писатель ≤1 новая страница/день (линии уже на scheduled-tasks),
  реоптимизатор 1–2 страницы/нед по списку аналитика.
- Пятница: техаудитор (скрипты + разбор).
- Разведчик: раз в неделю пополняет бэклог.

### Шаг 4 — передача ролей внешним исполнителям (по одной за раз)
Порядок: сначала разведчик → Gemini CLI (открытые данные, web-grounding),
потом писатель → Codex (тяжёлая HTML-сборка). После каждой передачи —
неделя наблюдения: качество не просело → роль закреплена за исполнителем
(отметить в брифе), просело → откат на Claude.

### Шаг 5 — передача рутины Гермесу (когда ритм устоялся)
Реоптимизатор и техаудитор — в крон Гермеса (инструкция в `docs/agents/`,
как `crimea-data-refresh.md`). Оркестратор остаётся за владельцем+Claude:
решения «что делать» не автоматизируем, пока KPI не стабилен месяц.

### KPI (смотрит аналитик, понедельно)
Клики и показы GSC/Вебмастер; число страниц в топ-10; доля топ-страниц,
обновлённых за последние 7 дней; новые страницы, вышедшие в индекс.

## Итоговая система: кто работает, какие модели, кто что делает

| Роль | Что делает | Исполнитель сейчас | Кому передаётся (шаг) | Ритм |
|---|---|---|---|---|
| **Оркестратор** | читает отчёт+бэклог, назначает задачи недели, следит за playbook | Claude Fable/Opus (сессия с владельцем) | Гермес-рутина пн 10:00 МСК (шаг 5+, не раньше) | пн |
| **Аналитик позиций** 🔴 | GSC+Вебмастер+Метрика → отчёт недели, кандидаты на реопт | Claude (Mac, браузер) | Гермес при выносе на API GSC; внешним — НЕЛЬЗЯ (секреты) | пн |
| **Разведчик семантики** | Wordstat/подсказки/конкуренты → бэклог тем, проверка против реестра | Gemini CLI `gemini-3-pro-preview` (фолбэк Claude) | DeepSeek-Flash / Гермес (шаг 4) | 1×/нед |
| **Писатель посадочных** | тема из бэклога → страница по playbook + реестр + sitemap + nav | Claude Sonnet-субагент | Codex (шаг 4); при подъёме seo-geo-engine — его конвейер | ≤1 стр/день |
| **Реоптимизатор** | свежесть/данные/CTA на страницах из списка аналитика | Claude scheduled-task | Гермес-крон (шаг 5) | 1–2 стр/нед |
| **Техаудитор** | sitemap/canonical/schema/битые ссылки/кеш — скрипты + разбор | скрипты + DeepSeek-Flash разбор | Гермес-крон (шаг 5) | пт |

Правила для всех ролей: git-протокол `HERMES.md §0` (intent в `AGENT_ACTIVITY.md`,
никаких autostash/reset в общем дереве, публикация из чистого клона origin),
нейтральный OSINT-тон, русские даты в UI, площадь ответственности — только
свои файлы из таблицы выше.

## Связь с проектом seo-geo-engine (Попов)

`wiki/projects/seo-geo-engine/` — отдельный проект (конвейер «Кибер SEO & GEO 2.0»,
12 агентов, 18-блочная анатомия статьи, статус planning с 2026-05). Здесь его НЕ
строим. Стыковка одна: бриф писателя держит совместимый контракт «тема из
бэклога → готовая страница», так что когда движок Попова поднимется, он встанет
на место писателя без перестройки остальной системы.
