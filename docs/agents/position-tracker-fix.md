# Position tracker: контаминация исправлена, VPS требует реконфига

Статус на 2026-07-11: P0 SEO-баг починен в репо, но VPS watchdog-рутина
пока не переключена — без этого баг вернётся на следующем прогоне.

## Что было сломано

`data/yandex-positions.json` — трекер позиций в поиске. Реально в нём
отслеживались только 2 запроса: `обслуживание итп` и `обслуживание итп Москва`.
Это ключи **чужого проекта** (logika-itp), не имеющие отношения к npz-tactical-map.
Трекинг позиций NPZ в поиске был полностью слепым.

Файл пишет внешняя VPS-рутина "watchdog" (коммиты `health: watchdog ...`,
несколько раз в день). Генератора этого файла внутри репо нет — рутина
читает список ключевых слов из **своего собственного конфига на VPS**, и
этот конфиг был указан на ИТП вместо NPZ.

## Что исправлено в репо

1. **`data/position-queries.json`** — новый источник правды: 40 реальных
   NPZ-запросов, отобранных из живых данных Wordstat (`seo/semantic-core.json`,
   `seo/wordstat-extended.json`, `seo/wordstat-regional.json`,
   `seo/popular-queries-extracted.json`) с маппингом на целевые URL из
   `data/seo-topics.jsonl`. Покрыты кластеры: бренд, Крым, дефицит, кризис,
   удары/НПЗ-список, объектные страницы заводов (Московский/Омский/Рязанский/
   Куйбышевский/Саратовский), регионы (Краснодар/Москва/Иркутск), радар/тревоги,
   АЗС. У каждого запроса есть `freq` (реальная частота Wordstat, где найдена)
   или `null` (editorial-подбор без прямой записи в источниках, но с живой
   целевой страницей).

2. **`data/yandex-positions.json`** — сброшен на чистый NPZ-скелет: те же
   40 запросов из `position-queries.json`, схема полей не менялась
   (`query`, `ads_position`, `ads_found`, `organic_position`, `organic_found`,
   `total_ads`, `total_organic`, `competitors_in_ads`, `ads_error`), все
   позиции `null`/`false` — ещё не измерено. `checked_at` = `2026-07-11T00:00:00+00:00`.
   Добавлено поле `_note` с объяснением сброса.

## Что должен сделать оператор VPS (Гермес)

**Watchdog-рутина, которая пишет `data/yandex-positions.json` (коммиты
`health: watchdog ...`), должна читать список запросов из
`data/position-queries.json` этого репо** (поле `queries[].query`), а не из
своего локального конфига на VPS. Пока это не сделано — при следующем
прогоне watchdog снова перезапишет файл ключами ИТП, и вся эта правка
обнулится.

Куда смотреть на VPS: конфиг/скрипт, который генерирует
`data/yandex-positions.json` и коммитит его с сообщением `health: watchdog`.
Найти в нём захардкоженный или неверно указанный список запросов
(сейчас — ИТП) и заменить источником на `data/position-queries.json` из
этого репозитория (синхронизировать через `git pull` перед запуском, как
остальные датасеты).

## Что должна содержать нормальная запись измерения

На каждый запрос — по факту прогона трекера:

- `query` — сам запрос (из `position-queries.json`)
- `engine` — `yandex` или `google` (сейчас трекается только Yandex; при
  добавлении Google — та же структура, отдельный файл или поле `engine`)
- `region` — регион поиска (сейчас не задан явно; watchdog должен
  зафиксировать регион XMLRiver-запроса, например `Москва` или `Россия`)
- `organic_position` — позиция в органической выдаче (число или `null`)
- `target_url` — целевой URL страницы (берётся из `position-queries.json`,
  чтобы сверять факт с планом — ранжируется ли именно та страница, что
  задумана)
- `date` — дата измерения (сейчас есть `checked_at` на уровне файла, но
  при переходе на историю по датам — на уровне записи)

Поля `ads_position`/`ads_found`/`total_ads`/`competitors_in_ads` — сохранить,
это уже работает и полезно (реклама конкурентов в выдаче по нашим ключам).
