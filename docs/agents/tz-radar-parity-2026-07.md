# ТЗ: расхождение карты БПЛА (/radar) с radar-map.ru — аудит и план фикса

Дата аудита: 2026-07-12, ~22:15 МСК. Статус: **аудит, код не менялся**. Ревью Codex (gpt-5.6-terra) пройдено 12.07 — правки внесены, см. раздел «Ревью Codex» внизу.

## Симптом

Оператор сравнил наш /radar с radar-map.ru (12.07 вечер): у них — десятки активных БПЛА-тревог и подсвеченные районы в Калужской/Тульской/Орловской/Брянской областях; у нас в этих областях — пусто (ни маркеров, ни строк в панели угроз).

## Ключевой факт

Источник данных у нас и у них **один и тот же**: `https://radar-map.ru/api/state`.
Цепочка: upstream → Vercel-прокси `api/radar-state.js` (live-фетч на каждый запрос, `s-maxage=60`, вырезает `pvo`) → `radar.html` фетчит `/api/radar-state` каждые `poll_interval_sec`. Задержка ≤60–120 сек. Параллельный VPS-cron пишет `data/radar-state.json`, но его читает только iframe-виджет `embed/radar.html` — главная карта им не пользуется.

**Свежесть данных — НЕ проблема.** Проверено live 12.07 22:08: `/api/radar-state` отдаёт события минутной давности. Расхождение целиком в рендере.

## Причины (по убыванию веса)

### RC1 — жёсткий whitelist `NPZ_REGIONS` (главная причина)

`radar.html:618-633` — `NPZ_REGIONS`: 14 регионов с НПЗ (Краснодарский край, ЛО, Ярославская, Москва+МО, Крым, Запорожская, ЛНР, ДНР, Севастополь, Волгоградская, Самарская, Саратовская, Ростовская и др.).

- `matchRegion()` (`radar.html:653-669`) возвращает `null` для всего вне whitelist + `REGION_ALIASES` (`radar.html:636-649`).
- Регион-маркеры НПЗ (`radar.html:984`) и **панель угроз** (`renderPanel`, `radar.html:1077`) итерируют только `Object.keys(NPZ_REGIONS)`.
- **Уточнение по ревью Codex:** city-маркеры whitelist'ом НЕ гейтятся — цикл `radar.html:1022-1055` рендерит все активные города в `cityCluster` без `matchRegion()`. Т.е. точки в Калужской/Тульской/Орловской/Брянской на карте появляются, но эти области **не попадают в панель угроз и не получают регион-маркеров/акцента** — на беглый взгляд «у нас пусто/не то, что на radar-map».

Противоречие продукту: сама страница (`/karta-bpla` и H1 на /radar) обещает «угрозы по регионам России», SEO-кластер — «карта БПЛА онлайн по регионам РФ». Whitelist «только НПЗ-регионы» противоречит собственному обещанию страницы.

### RC2 — заливка полигонов опирается на регион-флаги, которые у upstream пустые для тыловых областей

`regionThreatIndex` (`radar.html:949-972`) whitelist'ом не ограничен, но приоритетно читает `regions[имя].bpla|rocket|…`. В снапшоте 12.07 19:10 UTC у Калужской/Тульской/Орловской/Брянской обл. **все регион-флаги `false`**, хотя `source_text` содержит явные тревоги, а в `cities[]` активны 118 из 121 записи (`bpla: true`). Т.е. настоящий сигнал живёт на city-уровне; регион-уровень для тыловых БПЛА-тревог upstream не выставляет. City-цикл (`radar.html:957-963`) подхватывает флаг только если регион ещё не занят — фактическую подсветку этих 4 областей надо проверить визуально в рамках фикса.

### RC3 — пустой первый кадр (частично уже закрыт)

Live-проверка: при открытии /radar карта несколько секунд полностью серая (полигоны не подсвечены, маркеры не видны) до прихода `/api/radar-state` и geojson. **По ревью Codex:** loading-стейт в панели уже есть (`radar.html:555-560`, `renderState("loading", …)` `1197-1212`) — проблема только в том, что сама карта (полигоны) до загрузки выглядит пустой. Переформулировать как quality-улучшение, не как отсутствующую функциональность.

### RC4 — мусор в данных, который мы не фильтруем

Пример из live-ответа `/api/radar-state`: city-запись с ключом `калужская область|брянская область` — name «Калужская область», region «Брянская область», координаты Брянска (53.31, 34.28). Это артефакт парсинга upstream (сообщение упоминало два региона). Мы рендерим как есть → маркер «Калужская область» на территории Брянской. Нужна валидация: запись, у которой name = название региона ≠ region, отбрасывать или переклассифицировать в регион-уровень.

### RC5 — мелкие долги

- `bpla_icon_fade_sec` (10800 c) прокидывается прокси (`api/radar-state.js:73`), но в `radar.html` не используется. **Поправка по ревью Codex:** `bplaDim` при этом УЖЕ отражается визуально (`radar.html:1037-1042`, CSS opacity .45 на `:257`) — долг только в неиспользуемом fade_sec.
- Один глобальный stale-флаг на весь снапшот (`radar.html:947`, порог 20 мин) вместо per-region/per-city свежести.
- `feed: []` в VPS-снапшоте (`feed_count: 0`) — виджет embed живёт без ленты; проверить, отдаёт ли upstream feed вообще.
- Комментарий `radar.html:945` «restored: MiMo refactor dropped these» — логика свежести уже ломалась при автогенерации; при фиксе не трогать её без нужды.
- **Поправка по ревью Codex:** `data/radar-state.json` читает не только `embed/radar.html` — его читают и боты: `hermes/bot/poll.py:104`, `hermes/bot/threat_digest.py:86`, `hermes/bot/radar_alerts.py:254`. Схему файла не менять без проверки этих потребителей.

## Что предлагается (для исполнителя; код в этом ТЗ не пишем)

1. **Снять whitelist для отображения** (RC1): маркеры и панель строить по всем регионам/городам/районам из данных, а не по `NPZ_REGIONS`. `NPZ_REGIONS` оставить только как приоритизацию (НПЗ-регионы — вверху панели / акцентная иконка), не как фильтр. Это прямое требование оператора «наша карта должна соответствовать radar-map».
2. **Подсветка регионов от city/district-сигнала** (RC2): регион-флаг и city-флаги объединять через OR (максимум угрозы), а не «регион приоритетнее».
3. **Loading-стейт** (RC3): скелетон/надпись до первого успешного фетча.
4. **Фильтр мусорных city-записей** (RC4): name совпадает с названием субъекта РФ и не равен region → не рендерить как город.
5. Опционально (RC5): затухание иконок по `bplaDim`/возрасту события, per-запись свежесть.

Ограничения: ⛔ ПВО на карте не отмечать (поле `pvo` вырезается в прокси — не трогать). Нейтральный тон. Фронт-правки только из worktree + бамп `?v` у затронутых js/css + коммит сразу (git-sync hard-reset).

## Критерии приёмки

- Во время активной волны БПЛА в Калужской/Тульской/Орловской/Брянской обл. на /radar видны маркеры и строки панели по этим областям, полигоны подсвечены — картина сопоставима с radar-map.ru (одни данные → одна география).
- Маркер «Калужская область» с брянскими координатами не появляется.
- Первый кадр /radar не выглядит пустым.
- 13 НПЗ-регионов не потеряли текущее поведение (панель, приоритет).

## Ревью Codex (gpt-5.6-terra, 12.07.2026) — требования к исполнителю

Фактические поправки внесены выше (RC1/RC3/RC5). Дополнительные требования из ревью:

**Блокеры дизайна фикса:**
- Для не-НПЗ регионов нет координат и `info.objects`: `renderPanel()` (`radar.html:1101`) дереференсит `info.objects` из `NPZ_REGIONS` — карточка для произвольного региона требует отдельной модели (иначе throw). Регион-центроиды брать негде — либо считать из geojson, либо не рисовать регион-маркеры вовсе, ограничившись подсветкой полигона + city-маркерами.
- Не дублировать city-маркеры регион-центрами: цикл `cityCluster` (`1022-1055`, кластеризация `721-732`) уже рендерит все активные города — добавление регион-маркеров поверх раздует кластеры.

**Мажоры:**
- OR-объединение флагов делать по-флагово ДО `dominantType()` (приоритет `rocket → aviation → uab → bpla → fpv`, `radar.html:700-703`), а не «кто последний записал».
- Правило фильтра мусорных записей (RC4) — по канонизированным именам субъектов + несовпадению `region`, иначе можно выкинуть легитимные совпадения город/субъект.
- Geojson: 4 названные области есть в `russia-regions.geojson` (83 features, `radar.html:715-718`); при расширении географии валидировать нормализацию имён против всех трёх файлов (RF/новые территории/Крым).

**Дополнить критерии приёмки:**
- Контракт панели для не-НПЗ регионов: только активные или все, сортировка, лимит/виртуализация, чем заменить блок «Объекты».
- Детерминированная фикстура: сохранённый payload 12.07 (регион-флаги false + активные city-записи по 4 областям) как тест-кейс, не «ждать живую волну».
- Регрессия: `pvo` по-прежнему отсутствует во всех маркерах (политика прокси `api/radar-state.js:50-59`).
- Отдельные проверки путей рендера: city / district / region-only / route / sea / flight (`radar.html:1024-1027`; districts сейчас влияют только на агрегатное состояние, `925-943`).
- Измеримые состояния первого кадра: loading / geojson-fail / api-fail / retry / успешная замена loading-UI.
- Производительность при плотной волне: кластеры, мобильная панель, отсутствие дублей маркеров в одной точке (`1031-1033`).
