## 1. Посты: почему сумбур

**Корень: нет единого диспетчера.** Параллельно публикуют 4–5 несогласованных генераторов текста:

| Публикатор | Триггер (реальный, из `~/.hermes/cron/jobs.json` на VPS) | Формат |
|---|---|---|
| NPZ Morning/Afternoon/Evening Update | 08:00 / 14:00 / 20:00 UTC | свой; шлёт в **личку 609952529, не в канал** |
| NPZ MORNING/EVENING BRIEFING (`broadcast.py --briefing`) | 07:00 / 19:00 UTC | `compute_briefing_digest` — свой формат |
| NPZ PUBLISH → `broadcast.py` editorial + default | 08:30 / 14:30 / 20:30 UTC | `editorial_digest.py` (единственный по редполитике) + diff-дайджест |
| NPZ NEWSWATCH | почасовой fast-lane → канал `-1004491068477` | свой |
| RADAR ALERTS (`radar_alerts.py`) | каждые 10 мин | ok — единственный полностью рабочий механизм |

Итого подписчик получает 3–5+ пересекающихся постов в день в разном оформлении.

Ключевые факты:
- **«Молния» построена, но не подключена**: `strike_pipeline.py` + `radar_publish.py` (TIER1/TIER2) вызываются только с `--dry-run` (`agents/update-prompt-newswatch.md:8`). Срочного формата в проде фактически нет.
- Редполитика `docs/npz-posting-style.md` соблюдается 1 генератором из 4–5.
- Три отдельных состояния дедупа: `data/pipeline-state.json`, `~/.npz-bot/state.json`, editorial-state — один удар может выйти дважды.
- `~/.hermes/scripts/npz-*.sh` — **копии**, не symlink: правки в репо не долетают до исполняемых версий.
- Расписание: `agents/crontab.txt` — мёртвый артефакт эпохи claude-раннера; `hermes/crontab.hermes` — референс, не установлен; правда живёт только в `~/.hermes/cron/jobs.json`.
- Мешают и 429 от free-tier моделей (видно в `~/.hermes/logs/agent.log`) — часть сбоев публикаций не архитектурные, а квотные.

**Редакторские проблемы (DeepSeek, с примерами):** главное событие дня хоронится в 5-м абзаце; машинные штампы «Главное/Почему важно/Статус» без журналистской обработки; абзацы-простыни на 750+ знаков со сливом CNN/Reuters/биржи в одну кучу; «голоса людей» подменяются OSINT-рерайтом; повторные удары (Гвардейское 03.07 и 06.07) подаются как несвязанные события.

### План перестройки под целевую модель
1. Один источник правды расписания: `~/.hermes/cron/jobs.json`; `agents/crontab.txt` — в архив; HERMES.md — синхронизировать с реальностью.
2. Слить Morning/Evening Update + BRIEFING → **одна утренняя (07:00 UTC / 10:00 МСК) и одна вечерняя (19:00 UTC / 22:00 МСК) сводка**, одновременно в канал и на сайт (`/news/<дата>.html` — та же сущность).
3. Пополнение в течение дня: одна активная сводка; Telegram — `editMessageText` (или апдейты pinned-комментарием), сайт — блок `🔄 Обновлено HH:MM`, маркер сверху, максимум 3 апдейта/день.
4. Подключить «молнию»: убрать `--dry-run`, TIER1 — авто, TIER2 — кнопка подтверждения владельцу; **единый ключ дедупа** на все форматы (молния «занимает» слот «Главное» в следующей сводке).
5. Единый рендер-модуль (шаблоны/эмодзи/тон) для сводки, молнии и `gen-news.py` — редполитика соблюдается программно.
6. БПЛА-алерты (`radar_alerts.py`) не трогать — работают; задокументировать как отдельный канал.

Готовые жёсткие шаблоны трёх форматов (сводка 2000–2500 знаков, молния 3–5 строк, БПЛА-алерт ≤200 знаков) с примерами на реальных данных 06.07 — в отчёте DeepSeek (см. scratchpad `deepseek-posts-audit.md`, перенести в `docs/npz-posting-style.md` v2).
