# ТЗ: «Волна дронов» — real-time детектор → автостатья

**Проект:** npz-tactical-map · **Дата:** 12 июля 2026 · **Статус:** дизайн утверждён, ждёт плана
**Идея:** Серёга. Ловим всплеск активности БПЛА по radar-state → автоматически публикуем
статью-оповещение «Волна дронов идёт сейчас» с картинкой. Информируем население, ловим горячий
поисковый запрос («волна дронов сейчас / куда летят дроны»), заводим читателя.

Решения приняты в брейншторме (все — рекомендованные варианты):
- Публикация: **полный автопаблиш** (без ручного подтверждения).
- Раздел: **живая страница `/volna-dronov` + датированный архив** (не дубль в /news).
- Детектор: **средний порог** (≥25 городов / ≥4 региона, кулдаун 6ч).
- Картинка: **шаблонная inline-SVG-карточка**, мгновенно, $0 (не AI-генерация).
- Крон: **в существующую радар-цепочку** (`hermes/cron-radar-refresh.sh`).
- Монетизация: **лёгкий блок подписки на @BPLAlert_bot** + VPN-нудж по текущей схеме.

## 1. Сигнал (уже в наличии)

`data/radar-state.json` обновляется VPS-кроном каждые 1–5 мин. Поля по каждому из ~124 городов:
`bpla`, `bplaDim` (потухший/старый), `last_event_ts` (epoch сек), `region`, `lat/lon`.
Новый источник НЕ заводим — читаем то, что уже поллится.

**Яркий свежий город** = `bpla == true` И `bplaDim == false` И `(now − last_event_ts) ≤ 45 мин`.

Проверено на живом снимке 12.07 22:20 МСК: 49 ярких городов в 8 регионах (Тульская 11,
Московская 9, Орловская 7, Калужская 7, …) — это ровно текущая волна.

## 2. Детектор `agents/wave-detect.py` (новый, только stdlib)

Мимикрия под `agents/strike-candidates.py` (тоже детектор поверх radar-state, тоже не трогает
чужие данные). Запускается каждый радар-крон.

### Пороги (константы наверху файла)
```
RISE_CITIES   = 25    # волна начинается: ярких свежих городов ≥ 25 …
RISE_REGIONS  = 4     # … И регионов ≥ 4
FALL_CITIES   = 15    # волна кончается: ярких свежих городов < 15 (гистерезис — не мигать)
FRESH_SEC     = 45*60 # «свежесть» события города
COOLDOWN_SEC  = 6*3600# минимум между двумя РАЗНЫМИ опубликованными волнами
STALE_SEC     = 30*60 # стоп-гард: снимок старше — молчим
```

### Стоп-гард (защита от протухших данных)
Если `now − parse(radar-state.fetched_at) > STALE_SEC` → **выйти, ничего не делать**. Прямо
закрывает известную болячку: VPS-сбор падает (`claude` разлогинился и т.п.) → radar-state
замирает → без гарда мы бы вечно публиковали одну «волну». (См. `npz-vps-claude-logout-kills-agents`.)

### Машина состояний (хранится в `data/wave-state.json`)
```
quiet ── cities≥RISE И regions≥RISE_REGIONS И (now−last_published ≥ COOLDOWN) ──▶ active(new event)
active ── cities<FALL (сохраняется ≥1 крон-тик) ──▶ quiet (ставим ended_at событию)
active ── всё ещё волна ──▶ обновить peak_cities/peak_regions/region_list текущего события
```
- **Rising edge → публикуем:** создаём событие в `data/wave-events.json`, дёргаем `wave-cover.py`
  + `gen-wave.py`, коммитим (git-sync подхватит и запушит → Vercel деплой).
- **Пока волна идёт** (внутри кулдауна): НЕ плодим новую статью — обновляем `wave-state.json`
  (пик, регионы). Живая страница читает его клиентом и показывает свежие цифры без регенерации.
- **Falling edge:** дописываем `ended_at` в архивное событие, регенерим его снимок один раз.

### Файлы данных
- **`data/wave-state.json`** (перезаписывается каждый запуск, публикуется на сайт — читает JS живой страницы):
  ```json
  {"active": true, "cities": 49, "regions": 8,
   "region_list": ["Тульская область","Московская область", "..."],
   "started_at": "2026-07-12T19:00:00Z", "peak_cities": 52, "peak_regions": 9,
   "current_event_id": "2026-07-12-1900", "updated_at": "2026-07-12T19:20:00Z"}
  ```
  Когда тихо: `active:false`, `current_event_id:null`, плюс `last_event` (дата последней волны).
- **`data/wave-events.json`** — append-only архив (как `strikes.json`): `id`, `date`, `started_at`,
  `ended_at`, `peak_cities`, `peak_regions`, `region_list`, `published_at`. Раз попав — не теряется.
  🔴 Добавить в pre-commit shrink-guard (как у strikes.json, `npz-strikes-shrink-guard`): архив
  не должен усыхать.

## 3. Картинка `agents/wave-cover.py` (новый) — Codex-first, SVG-фолбэк

**Контракт для gen-wave:** `build_card(event: dict, out_dir: str) -> {"inline_svg": str, "png_path": str|None}`.
`event` = {date, cities, regions, region_list, started_at}.

Порядок (по решению Серёги):
1. **PRIMARY — Codex-генерация картинки** через существующую NPZ-cover-цепочку (`build-covers.py`,
   env `NPZ_COVER_BACKENDS`, Codex@VPS→Codex@Mac, `feedback-npz-covers-via-codex`). Сюжет:
   **летят дроны + дата + где летят** (регионы/города волны, крупная подпись с датой). Отдаём в
   `png_path`.
2. **Жёсткий таймаут 5 минут** на Codex. Не уложился ИЛИ упал/пусто → **фолбэк на inline-SVG**.
3. **FALLBACK — SVG-карточка** 1200×630: упрощённый контур РФ (из `data/russia-regions.geojson`,
   грубо), подсвеченные регионы, иконка дрона, крупно «ВОЛНА ДРОНОВ», дата рус-форматом
   («12 июля 2026», helper `rusDate()` в `app.js`), список регионов + «N городов». Отдаём в
   `inline_svg`, мгновенно, 0 зависимостей.

На странице всегда есть чем показать: если Codex дал PNG — `<img>`, иначе inline `<svg>`. Публикацию
страницы обложка **не блокирует** (SVG-фолбэк гарантирован). OpenRouter — только если Codex-бэкенды
недоступны (последний рубеж, `NPZ_COVERS_ALLOW_OPENROUTER`, кап — `npz-openrouter-key-location`).

## 5. Текст статьи — детерминированный шаблон (без LLM в автопаблише)

Автопаблиш 24/7 + нейтральная OSINT-гвардия (`npz-neutral-osint-guard`) ⇒ текст собирается по
шаблону из радар-фактов, НЕ пишется LLM (нет галлюцинаций, мгновенно, всегда нейтрально).
Факты объективны: сколько городов, какие регионы, во сколько началось.

Шаблон (нейтрально, без алармизма и без атрибуции сторон):
> «По данным публичного мониторинга radar-map.ru, вечером 12 июля 2026 фиксируется повышенная
> активность БПЛА: отметки в {N} населённых пунктах {M} регионов — {список регионов}. Это
> предварительные данные мониторинга, не официальная сводка. Следите за официальными
> оповещениями МЧС и региональных властей.»

Дисклеймер обязателен. Тон — как в `/radar`. Sanitize не нужен (внешнего вербатима нет).
LLM-полировка текста — **отложенный апгрейд**, не в MVP.

## 6. Врезка в крон (VPS, Гермес)

В `hermes/cron-radar-refresh.sh` — одна строка после `update-radar-state.py`, перед `git-sync`:
```bash
python3 agents/update-radar-state.py
python3 agents/wave-detect.py        # ← НОВОЕ: детектит волну, на rising edge генерит статью+карточку
python3 agents/healthcheck.py
bash agents/git-sync.sh "..." "radar-state"   # существующий git-sync подхватит wave-* файлы и запушит
```
Публикация = push → Vercel авто-деплой. Отдельный джоб не нужен: детектор бесполезен без свежего
radar-state, живёт в той же цепочке. Пуш анонимным аккаунтом — по текущему рецепту
(`GH_TOKEN=$(gh auth token -u volobuevaleksand7-hue)`, `npz-local-push-auth`); на VPS git-sync уже
настроен на нужный remote.

## 7. Тест (обязательный runnable-чек детектора)

`agents/wave-detect.py` под `if __name__ == "__main__"` или `test_wave_detect.py` — assert-чек
машины состояний на синтетических снимках (без сети, stdlib):
- 49 ярких/8 рег, свежий снимок, кулдаун истёк → **publish=True**, state.active=True.
- Тот же вход через 10 мин (кулдаун активен) → **publish=False**, обновлён peak, новой статьи нет.
- Города упали до 10 → active=False, у события выставлен ended_at.
- Снимок старше 30 мин (STALE_SEC) → детектор молчит (publish=False, state не тронут).
- 20 ярких / 3 региона → не волна (ниже RISE).
