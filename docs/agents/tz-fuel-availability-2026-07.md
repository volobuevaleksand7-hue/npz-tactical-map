## 0. Зачем

Боль клиента, заходящего на карту АЗС: увидеть **реальную ситуацию** — где сейчас есть бензин/дизель, где очередь, где закрыто. Сейчас наша карта показывает только OSINT-оценку по сети/региону, а не факт по конкретной колонке. Конкурент gdebenz.ru даёт реальные краудсорс-статусы — это его главное преимущество. Задача: дать нашему юзеру реальную картину, **не став вечным паразитом на чужой инфраструктуре и не подорвав доверие** (доверие — единственный актив проекта на чувствительной теме ударов).

## 1. Честная рамка (принять до старта)

- **Внешний источник = временные строительные леса, а не фундамент.** Математика явки жёсткая: чтобы наши отметки перекрыли ~100 свежих чужих статусов в городе, нужно ~1250–6700 гео-релевантных заходов/день именно там. При 1000 DAU это 15–80 своих отметок/день против 9609 станций — капля. Без явных graduation-метрик и дедлайна «затравка» превращается в постоянную зависимость.
- Поэтому: **свой краудсорс — первичен и строится с первого дня**, внешний слой — вторичен, атрибутирован и отключаем рубильником.
- **Цена ошибки высокая:** ложное «есть бензин» под ударами = пустая/рискованная поездка, и винят НАС, не источник. Свежесть и честная маркировка важнее полноты покрытия.

## 2. Архитектура песочницы

- Страница `/karta-azs-lab` — точная копия прод-страницы АЗС, `noindex,nofollow`, в robots.txt `Disallow: /karta-azs-lab`, не в навигации и sitemap.
- Вся новая логика — в `azs-lab.js` (overlay поверх нетронутого `app.js`). Прод `/karta-azs` и `app.js` не меняются, пока фича не доведена и не принята.
- Промоушен в прод (см. §9) — отдельным шагом после теста.

## 3. Источники данных слоя АЗС (фактическая структура на 21.07.2026)

Слой АЗС собирается из трёх слоёв данных: расположение (база) + старые свои OSINT-источники (регион/сеть) + новые живые люди (per-station). Ниже — что реально есть и что кормит карту.

### 3.0. База — расположение станций
- `data/azs-stations.json` — **9609 точек из OpenStreetMap** (координаты + бренд, без статуса). Обновляет `agents/refresh-osm-stations.py`. Это «где стоят заправки».

### 3.1. СТАРЫЕ свои OSINT-источники (сейчас в проде; уровень РЕГИОН/СЕТЬ, не колонка)
- **Наличие** — `data/fuel-availability.json` (**88 регионов × сети**: `level`, лимит л, цены, биржа СПбМТСБ, `note`, `updated`). Заполняет OSINT-агент `agents/update-prompt-availability.md`: WebSearch по горячим темам + ротация 12 самых старых регионов/прогон. 🔴 **Именно этим СЕЙЧАС красятся маркеры станций** (оценка по сети/региону).
- **Голоса водителей** — `data/fuel-voices.json` (**142 цитаты**, панель «Комментарии»). Агент `agents/update-prompt-voices.md` из двух каналов: **(A) публичный Telegram через веб-зеркало** `t.me/s/<channel>` (Crimeanwind, chp_krd, RndCHP, nefte_baza, oil_capital, baza, shot и т.п.) + **(B) WebSearch/СМИ/регио-медиа**. Разбивка источников: СМИ 36, Telegram 32, WebSearch 23, регио-медиа.
- ⚠️ Природа этих данных: это **оценка по региону/сети и агрегированные цитаты**, собранные **пачками (агент, лаг часы-сутки)** — НЕ статус конкретной колонки в реальном времени. Telegram здесь уже используется, но как ручной OSINT-разбор, не поток от юзеров.

### 3.2. НОВОЕ — люди онлайн (наш краудсорс, Ц2, ✅ в песочнице)
- **«Я тут»** — `api/azs-votes.js` + Upstash Redis. Живой пользователь тапает статус **по КОНКРЕТНОЙ станции в реальном времени** (есть/нет/очередь/лимит). Свежее окно 6ч, агрегат «👥 За 6 ч: …». Первый источник с точностью **до станции и до минуты** — то, чего старые слои не дают.

### 3.3. ПЛАНИРУЕТСЯ (Ц6) — Telegram-чаты автоматом
- Публичные районные **TG/MAX-чаты через `agents/collect.py` + `data/sources.json`** (пайплайн уже есть для ударов) — структурный поток, а не ручной агент. Требует новой способности: вытащить из прозы чата «станция + топливо + статус + время» (R&D). Кросс-проверка с «Я тут».

### 3.4. gdebenz.ru (внешний) — НЕ источник
- Решение принято: **не bulk-скрейпить** (их ToS запрещает автоскрейп + DDoS-Guard + SPOF + репутация). Максимум — спрос-драйв как отключаемые леса и только после запроса о партнёрстве/фиде (§8). По умолчанию в структуру НЕ входит.

### Как это блендится (целевая модель)
1. **Фон/затравка** — старая региональная оценка (`fuel-availability.json`), пока живых отметок мало.
2. **Поверх** — живые per-station «Я тут» + (позже) агрегированные TG-чаты, с **приоритетом свежести** и честной атрибуцией источника.
3. Голоса (`fuel-voices.json`) остаются контекстом в попапе/панели.
4. Внешний конкурент — не источник.
Правила блендинга (приоритет свежести, раздельные `observed_at`/`fetched_at`, conflict-resolution, graduation) — в §4–5.

## 4. Guardrails (обязательны, без них не публиковать)

- **Раздельные метки времени:** `observed_at` (когда человек видел) / `fetched_at` (когда мы скачали) / cache-timestamp. На карте показываем `observed_at`, а не момент фетча — иначе протухшее выглядит свежим.
- **Явные UI-состояния статуса:** свежо → протухает (через N часов «неподтверждено») → исчезает; отдельно «источник недоступен» ≠ «нет отчётов». Не сваливать всё в один цвет и не держать вечно зелёным.
- **Различать типы:** «нет бензина» / «нет конкретного топлива» / «очередь» / «лимит» / «закрыта» — не один красный.
- **Приватность «Я тут»** (тема ударов!): не логировать IP, retention отметок, огрубление координат, честное определение «анонимно» на VPS. Это не мелочь.
- **Anti-abuse:** localStorage-`cid` сбрасывается и скриптуется — репутация репортёра, дедлимит, защита от координированного вброса.
- **Модель доверия:** дедуп станций (сопоставление gdebenz `usr_*`-точек с нашим OSM-слоем по гео+бренду) — грязно на практике (соседние станции, стороны дороги, дрейф координат, франшизы). Нужна очередь ручного ревью неразрешённых матчей + confidence + persistent override-таблица. Форс-матч на слабых данных = «есть бензин» на неправильной точке.
- **Приоритет источников и conflict resolution:** правила, чей статус побеждает; проверка независимости перед тем как «совпадение двух источников» засчитывать за подтверждение (оба могут пересказывать один исходный отчёт).
- **Мониторинг:** fetch success rate, challenge rate, возраст кэша, число несматченных станций, коллапс покрытия, экспозиция протухших данных → алерт.
- **Рубильник источника:** выключаемый одним флагом; но честно знать, что при выключении gdebenz до набора своей массы карта фактически пустеет (скрытый SPOF). Регулярный drill «источник выключен — что реально видно».

## 4.5. Слияние статуса на маркере: живые отметки vs региональная оценка

**Принцип:** один маркер = один цвет, но **источник всегда виден**. Живая per-station отметка бьёт региональную оценку, ЕСЛИ она свежая и её достаточно. Иначе — фон-оценка по региону/сети.

**1. Что считает сервер (расширить `api/azs-votes` GET).**
Сейчас агрегат отдаёт counts + `last_observed_at`. Добавить **взвешивание по свежести** (у сервера есть timestamp каждой отметки в hash `av:{id}`):
- вес отметки = f(возраст): свежая ≈1, к концу окна 6ч → ~0 (напр. линейно `w = 1 − age/6ч` или экспонента);
- по каждому статусу — сумма весов; `top_status` = макс. сумма (НЕ сырой count);
- `confidence` 0–1 = f(суммарный вес + согласованность): растёт с числом свежих независимых `cid` и с единодушием, падает при раздрае;
- 🔴 почему свежесть, а не количество: заправка пустеет за час — «нет» 5 мин назад важнее «есть» 3ч назад.

**2. Правило выбора статуса (клиент, по станции):**
```
если live.count ≥ MIN_VOTES и live.confidence ≥ CONF_MIN:
    статус = live.top_status         // источник = живые отметки
иначе:
    статус = регион.level            // источник = оценка по региону/сети
```
Пороги `MIN_VOTES`/`CONF_MIN` — калибровочные (подбор на реальных данных): старт `MIN_VOTES=1`, `CONF_MIN` низкий (1 per-station факт всё равно точнее регион-догадки, но помечаем «1 водитель»). Поднять при спаме/шуме.

**3. Конфликт живых отметок (одни «есть», другие «нет»):**
- победитель — по взвешенной сумме (recency), не по большинству голов;
- near-tie (веса близки) → показать самую свежую + пометка «мнения расходятся».

**4. Цвет маркера (единая шкала наличия):**
- есть / `calm` → зелёный
- очередь / лимит / `strained` / `limited` → жёлтый
- нет / `severe` → красный
- нет ни живых, ни оценки → серый

**5. Провенанс всегда виден (доверие = актив).** В попапе — ОБЕ строки раздельно и подписанно:
- 👥 Живые отметки: N за 6ч, последняя X мин назад, «сообщения водителей, не гарантия»;
- 📊 Оценка по сети/региону: `<level>`, обновлено `<дата>`.
На маркере — маленький признак «live» (точка/кольцо) vs обычный контур «оценка». 🔴 Никогда не сливать в один безымянный цвет — юзер должен видеть, на чём статус, особенно если живое и оценка расходятся.

**6. Затухание/откат.** Живой сигнал вышел из окна 6ч → маркер откатывается на региональную оценку; попап: «живых отметок нет, показываем оценку по региону».

**7. TG-чаты (Ц6) — тот же слот, что live, но ниже приоритетом.** Когда появятся: приоритет **живые «Я тут» > TG-чаты > регион**. Кросс-подтверждение (живое + TG совпали) → выше `confidence`.

## 5. Graduation-метрики (задать ДО интеграции внешнего слоя)

Момент, когда gdebenz можно гасить, определяется числами, а не «на глаз»:
- доля первый-парти данных в видимых статусах;
- уникальных репортёров на станцию;
- медианный возраст наблюдения;
- correction rate, conflict rate;
- минимальное покрытие при выключенном gdebenz.
Плюс **дедлайн**: если к дате X доля своих < порога — пересматриваем стратегию, а не продлеваем зависимость молча.

## 10. Фаза 2: общий бэкенд для «Я тут» (Upstash Redis)

v1 виджета «Я тут» (готов, в песочнице) хранит отметку ТОЛЬКО на устройстве. Чтобы отметки стали **общими** (водитель видит, что отметили другие) — нужен write-стор. Это первый-парти актив, независимый от gdebenz.

**Выбор стора:** Upstash Redis — бесплатный тариф (~10k команд/день на старте), REST из Vercel-функции, встроенный TTL под «протухание», portable, приватно. Альтернатива — тот же Upstash через Vercel → Storage (Marketplace), разница только в UI.

**Действие владельца (один раз, ~10 мин):** включить Upstash (Vercel → проект → Storage → Marketplace → Upstash → Redis; либо upstash.com → Create Redis → вписать `UPSTASH_REDIS_REST_URL` + `UPSTASH_REDIS_REST_TOKEN` в env проекта Vercel, окружения Production+Preview). 🔴 Токены агенту не передавать — функция читает их из `process.env`.

**Что строит агент:**
- `api/azs-votes.js` (serverless, по образцу `api/radar-state.js`): `POST` `{station_id,status,cid}` → запись голоса; `GET ?ids=…` → агрегат по станциям `{count, breakdown:{yes,no,queue,limit}, top_status, last_observed_at, confidence}`.
- Модель Redis: hash `av:{station_id}` = `cid → {status, observed_at}` с TTL (~6ч), старьё выпадает само; агрегат и `confidence` (растёт с числом уникальных cid, падает с возрастом) считаются на чтении.
- Anti-abuse: 1 голос на cid/станцию в N минут (`SET NX` TTL) + общий rate-limit по cid.
- Приватность (§4): не логировать IP, не писать координаты; только `{status, ts, cid}`, cid — случайный клиентский.
- Провязка `azs-lab.js`: при тапе — localStorage + `POST`; при открытии попапа — `GET` агрегат «👥 N водителей за 6ч: …» над своей отметкой; **рубильник** — при недоступном API падаем на device-local, карта не пустеет (нет SPOF).
- Разделять `observed_at` (когда видел) и `received_at` (когда пришло); показывать возраст наблюдения, не фетча.

**Тест:** сначала в песочнице (Upstash работает и в Preview) — POST/GET, агрегация с двух разных cid, rate-limit, TTL-затухание, откат при выключенном API. Только потом — на промоушене (§9) — в прод.

## 11. ЧТО НЕ ДЕЛАТЬ

- ❌ Bulk-скрейпинг закрытого `/api/` gdebenz за DDoS-Guard и republish их статусов как «своих живых». Против их ToS, хрупко (гонка за сменой формата), SPOF на чужой инфре, репутационный удар. Дорого в поддержке — противоположность «дёшево».
- ❌ Показывать протухшие данные без пометки, вечно-зелёные статусы, «есть бензин» без возраста наблюдения.
- ❌ Форс-матч станций без ручной очереди ревью.
- ❌ Раскатывать рост-приёмы поверх ещё хрупкого слоя данных — усиление blast radius.
