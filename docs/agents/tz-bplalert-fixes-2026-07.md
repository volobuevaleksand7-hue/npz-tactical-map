## Контекст (даёт оркестратор каждому сабагенту дословно)

@BPLAlert_bot — публичный анонимный алерт-бот проекта «Топливный фронт РФ»
(npz-tactical-map.vercel.app), аккаунт Erhan08263. Жалоба владельца: **бот шлёт
только угрозы (радар), не шлёт удары в прямом эфире, и не соблюдает интервалы.**

Реальная архитектура на Hermes VPS (важно — доки `docs/agents/bpla-alert-bot.md`
устарели, там описан старый `poll.py`, но в проде другое):

- **Живой бот** = systemd-демон `npz-bpl-bot.service` → `hermes/bot/poll_bpl.py`
  (python-telegram-bot, long-polling). База: `/root/.npz-bot-bpl/subscribers.json`
  (~22 подписчика). `Environment=NPZ_BPL_DIR=/root/.npz-bot-bpl`, `NPZ_REPO=/root/npz-tactical-map`.
- **Зомби** = cron ежеминутно гоняет старый `hermes/bot/poll.py` с
  `NPZ_BOT_DIR=/root/.bpla-alert` (там 0 подписчиков) на **том же токене** →
  два конкурирующих `getUpdates`, в логе `HTTP Error 409: Conflict`.
- **Рассыльщик угроз** = `hermes/bot/radar_alerts.py --send`, но cron запускает его
  тоже с `NPZ_BOT_DIR=/root/.bpla-alert` (пусто) → каждые 10 мин честно шлёт 0 сообщений.
  Реальная база `/root/.npz-bot-bpl` рассылалась вручную ~раз в сутки.
- **Пять папок данных** на один бот: `.bpla-alert`, `.bpl-bot` (симлинки в `.npz-bot/`),
  `.npz-bot-bpl` (канон), `.npz-bot/bpl`, `.npz-bot`. Разные `radar-alert-state*` с
  разными датами последней отправки.

Регионы НПЗ и структура подписчика (`alerts: {enabled, regions:[], interval_min,
threats, attacks}`) — см. `poll_bpl.py`. Токен: `/root/.npz-bot-bpl/token`
(симлинк на `/root/.npz-bot/token-bplalert`), в git НЕ коммитить.

Общие правила проекта: нейтральный OSINT-тон, время в МСК (UTC+3), push в прод —
точечным `GH_TOKEN=$(gh auth token -u volobuevaleksand7-hue)`, дерево коммитить сразу

## Task 1 — Настройки интервала и типов уведомлений не сохраняются (🔴 главный баг)

**Файл:** `hermes/bot/poll_bpl.py`

**Симптом:** пользователь жмёт «30м» — бот отвечает «⏱ Интервал: 30 мин», но в файл
пишется старое значение. Отключить/включить «Угрозы»/«Атаки» невозможно. Интервал
у всех навсегда 60 мин. Именно отсюда «не соблюдает интервалы».

**Root cause:** паттерн двойной загрузки. В хендлерах сначала `subs = load_subs()`,
потом `sub = ensure_sub(chat_id, ...)`. Но `ensure_sub` (строки ~123-138) **внутри
себя заново** делает `load_subs()` и `save_subs()`, возвращая объект из своей копии.
Правки вносятся в `sub` (вторая копия), а финальный `save_subs(subs)` сохраняет
первую — нетронутую. Затронуты:
- `on_timer_callback` (~382-395): `subs = load_subs()` + `sub = ensure_sub(...)` → `sub["alerts"]["interval_min"] = …` → `save_subs(subs)`.
- `on_alerts_callback` (~495-500): то же с полем threats/attacks.
- `cmd_interval` (~411-428): то же с interval_min.

Работает только выбор регионов — потому что `set_sub_regions` (~168-173) читает и
пишет один и тот же `subs`, не зовя `ensure_sub`.

**Требования:**
1. Устранить рассинхрон: правка настройки должна попадать в тот объект, который
   реально сохраняется. Каноничный способ — привести три хендлера к паттерну
   `set_sub_regions`: `subs = load_subs()` → `sub = subs["subscribers"].setdefault(str(chat_id), <default>)` → мутировать `sub` → `save_subs(subs)`. Либо
   отрефакторить `ensure_sub`, чтобы он не делал собственный save и работал по
   переданному `subs`. Выбор за реализатором, но дефолт подписчика (структура
   `alerts`) и идемпотентность `setdefault(threats/attacks=True)` должны сохраниться.
2. Не сломать существующие вызовы `ensure_sub` в `cmd_start`, `cmd_interval` (там он
   создаёт подписчика при первом контакте).

**Критерии приёмки:**
- После «/interval 30» → перечитать `subscribers.json` → `interval_min == 30`.
- После нажатия кнопки «⏱ 10м» → в файле `interval_min == 10`.
- После тумблера «💥 Атаки» → в файле `attacks` инвертировался и **сохранился**.
- Регионы по-прежнему сохраняются (регресс не введён).

**TDD:** `hermes/bot/test_poll_bpl_persist.py` — юнит без Telegram: сохранить
tmp `subscribers.json`, вызвать переработанную логику (вынести чистую функцию
`apply_interval(subs, chat_id, val)` / `apply_toggle(subs, chat_id, field)` если
удобно) и проверить `assert` перечитанного значения. Один прогон, без моков сети.

## Task 5 — Устойчивость рассылки `radar_alerts.py` (🟡)

**Файл:** `hermes/bot/radar_alerts.py`

**Проблемы:**
1. `build_notifications` (~176-177) ставит `last_sent = now_ts` в state ДО фактической
   отправки, а `main` сохраняет state независимо от результата `send_message`. При
   403 (заблокировал) / 429 (rate-limit) сообщение теряется, но `last_sent` уже сдвинут
   → следующее напоминание отложено, а «new» не повторится.
2. Не-HTTP ошибка (сетевой таймаут в `send_message`) не поймана → весь прогон падает
   до `jsave(next_state)`. При следующем запуске state старый → **дубликаты всем**.
3. Джиттер интервала: порог `now - last_sent >= interval*60` при cron раз в 10 мин
   сдвигает «10м»→10-20 мин, «30м»→30-40. Мелочь, но фиксится допуском.

**Требования:**
1. Коммитить `last_sent` в state только для chat_id, которым сообщение реально ушло
   (`resp.ok`). Для упавших — оставлять прежний `last_sent` (не терять напоминание).
2. Обернуть отправку в try/except по широкому классу сетевых ошибок (URLError/timeout),
   логировать и продолжать цикл, а не падать. state сохранять всегда (частично обновлённый).
3. (Опц.) Допуск джиттера: `now - last_sent >= interval*60 - CRON_SLACK` (CRON_SLACK=60с),
   чтобы «10м» не превращались в 20.

**Критерии приёмки:**
- Юнит: при `send_message`, кидающем HTTPError 403 для chat X, — state для X не
  двигает `last_sent`; для успешного Y — двигает.
- Юнит: `timeout` на одном chat не мешает остальным получить сообщения в том же прогоне.

**TDD:** `hermes/bot/test_radar_alerts_send.py` — подменить `send_message` заглушкой,
раздающей ok/HTTPError/timeout, проверить итоговый state. Расширить существующий
`test_radar_alerts.py`, не плодить.
